2638 On Unix, the directory is not synced. In rare circumstances, transaction log files may be lost after a crash.

Other Notes and Changes

  • The '-daemon' option when starting RDM Servers on Unix has been replaced with '-start'.
  • The TFS type can now be selected at runtime, see TFS Configurations for details. In addition, the multiple TFS libraries have been consolidated into a single library.
  • Significant changes have been made to the SQL Virtual Tables API. Please carefully read through the "Using Virtual Tables to Access Any Data" chapter in the SQL User's Guide for details.

Database Compatibility

Databases created with earlier versions of RDM are compatible with the current RDM 11.0. Although a new Database Dictionary (DBD) format has been introduced, the current RDM still recognizes the old format, and can still generate DBD files in the old format. A new Database Dictionary (DBD) format was necessary to support long filenames and Unicode, since the names of all data and key files in a database are held in the DBD file. The current RDM version supports the following DBD formats:

Database
Version		 Description
V3.00 Used by all versions of RDM from 3.0 to 4.5, with ANSI data and key filenames up 48 characters long
V3.01 Previously only recognized in RDM/Vx 4.9.3, with VxWorks-specific DDL extensions
V5.00 ANSI filenames up to 256 characters long, and v3.01 extensions
U5.00 Unicode filenames up to 256 characters long, and v3.01 extensions
V6.00 V5.00 database extended with varchar support
U6.00 U5.00 database extended with varchar support

If you do not use any of the more recent features, the ddlp program supplied in the current RDM will generate DBD files in the old V3.00 format.

If you have written any programs that read the RDM database dictionary (DBD file) directly, be sure you are using the correct FILE_ENTRY structure in the dbtype.h file to interpret the DBD file.

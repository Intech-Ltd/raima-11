UNIX/Linux/Mac OSX Platform Release Notes

Introduction  

This PS (Platform-Specific) Guide focuses on the UNIX/Linux/Mac OSX operating system for RDM. This section explains:

  • How to set up the UNIX development platform
  • How to build RDM & examples
  • Other issues specific to Unix/Linux

Development Environment 

The build scripts for RDM  examples and source assume that the paths and environment variables for the desired compiler have already been setup.

RDM  Setup Script Files

. ./rdm.sh [Target] [TFS] [Options]

The script must be sourced when invoking, by prepending a ". " before the script name, in order to correctly set environment variables to the current shell.

Target

-32

Specify the target platform architecture as 32bit

-64

Specify the target platform architecture as 64bit

-lnxarm

Specify cross compile for Linux on the ARMv7 instruction set (Cortex Series)

-vx6x86

Specify cross compile for WindRiver VxWorks on the Intel x86 processor

-neux86

Specity cross compile for QNX Neutrino on the Intel x86 processor

-neuarm

Specity cross compile for QNX Neutrino on the ARMv7 instruction set (Cortex Series)

The architecture default will vary depending on the current platform.

TFS

-tfsr

Use the Client/Server TFS version (Default)

-tfss

Use the Standalone TFS version

-tfst

Use the Direct Linked TFS version

If using multiple shell windows, changing the current TFS will affect all shells using that install directory.

Options

-gnutools

Use the GNU toolchain when compiling on AIX, HPUX, or Solaris

The default compiler will be the GNU compiler toolchain for all platforms except:

Solaris (Sun Studio Development Environment)
HPUX (HP C/aC++  Developer's Bundle)
AIX (VisualAge C++)

Building RDM (source package) or Examples

  1. Install the software following the Unix Install Guide. If building source, ensure you are installing the source package.
  2. From a terminal window session:
>cd [installDir]
>. ./rdm.sh [options]
>k

As part of the build step (for source or examples), a tfserver is automatically started before and stopped after. Shutdown any running tfservers before starting the build.

If a source package is installed, this will build the entire source if source files are modified. If source is not present, only the examples will be built.

All RDM makefiles are designed to be invoked by the KMake utility, which is included in the RDM Installation. The correct command is aliased to 'k' for convenience.

In-Memory Issues

Unix shared memory segments do persist beyond the lifetime of the process that created them. RDM can use either of two alternative sets of routines for creating and maintaining shared memory segments: Posix or System V. Which of these alternatives is used depends on whether the pre-processor symbol NO_POSIX_SHMEM is defined when RDM is compiled (see file pspunix.h). For deliveries that include object code, the following defaults apply:

HPUX, Linux, FreeBSD: NO_POSIX_SHMEM defined, System V shared memory used
All other Unixes: NO_POSIX_SHMEM not defined, Posix shared memory used

For systems where System V shared memory is used, the segments created by RDM can be listed with the system command ipcs, and removed with ipcrm. Note that the system limits the number of System V shared memory segments a process can create. If your database grows large, or has a lot of files, RDM may exceed this limit, causing a S_BADWRITE (-919) error or a S_LOGIO (-913) error. If this happens, try increasing the values of the following kernel parameters: 

kern.ipc.shmmni
kern.ipc.shmseg
kern.ipc.shmmax
kern.ipc.shmall

On Mac OS X:

kern.sysv.shmmax
kern.sysv.shmmin
kern.sysv.shmmni
kern.sysv.shmseg
kern.sysv.shmall

Please see http://support.apple.com/kb/HT4022 for more information.

ARM Support

RDM also includes support on Linux and QNX Neutrino for ARM using the ARMv7 instruction set. This includes the Cortex Series- the Cortex A8 and Cortex A9.

There is a known issue on ARM platforms with the handling of de-normalized floating point values. It is advised not to use a floating point value in an index on ARM based platforms.

Secure Linux Environment (SELinux) Support

The default build option for RDM does not support the "Enforcing" security level on the Linux platform. A system error similar to the following will be encountered: 

librdmpsp11.so: cannot restore segment prot after reloc: Permission denied

This problem is caused by text relocations required in the RDM code. It can be solved by compiling RDM to produce position independent code by adding the -fPIC option to the compiler command line, in syslnx.mak. Unfortunately, adding this option may slow RDM down by up to 15%.

Another way of resolving the issue is to compile the application using the static libraries included with the product.

A.1 Compilation Phase

Warnings are provided during the compilation phase of db_REVISE when a potential error condition exists or when db_REVISE will not be able to perform a specified action.  Compilation warnings will be printed by default.  The -c option may be used to suppress the printing of warnings (i.e., -c0).  Errors during the compilation phase will terminate the compilation.

A.1.1 Warnings

A compilation warning has the following form: 

dbrev compilation warning: message

Individual compilation warning messages are listed below.  Each message appears in an example box and is followed by an explanation of its meaning.  

conversion to sorted set not allowed, set setname

A destination set is ordered ascending or descending, but the source set order is different.  db_REVISE cannot sort a set during revision.  

'convert field' statement ignored for field fieldname because both 
source and destination must be character type

A convert field statement specifying centering, left justification, or right justification has been used between source and destination fields where the source field type is not character.  (See the convert field statement in section 3.4.2, "Field Changes.")  

'convert field' statement ignored for field fieldname 
because it is a new field

A field can only be converted when it is contained in the source database.  Only new fields may be initialized.  

'convert field' statement ignored for field fieldname because 
neither source nor destination are character type

A convert field statement specifying an sprintf/sscanf format must be used between a character type field and a non-character type field.  See the convert field statement in section 3.4.2, "Field Changes."  

'convert field' statement ignored for field fieldname because 
source and destination are both character type

A convert field statement specifying an sprintf/sscanf format must be used between a character type field and a non-character type field.  See the convert field statement in section 3.4.2, "Field Changes."  

destination set type setname is new because source set type of same 
name is combined

A combine set statement has named setname to be combined with a set with another name.  If setname is used in the destination database, it will be treated as a new set type because all existing connections are now a part of the combined set.  

field type fieldname is used in a 'sorted by' clause, and should 
not be converted

A field used to order a set can be converted, but if a conversion is made that changes the sequence of the set, the set will become out of order.  (See section 3.4.2, "Field Changes.")  The negative effects of this revision may not be experienced until the database is updated.  

field type fieldname is used in a 'sorted by' clause, and 
should not be shortened

Field fieldname is a character string and its length has been shortened in the destination database.  This will probably cause the sorting sequence of the field to be changed.  (See section 3.4.2, "Field Changes.")  The negative effects of this revision may not be experienced until the database is updated.  

set ordering changed from 'order' to 'order' in setname

When the ordering of a set is changed, the physical order of the set members will not be changed during the revision.  This message is more of a reminder than a warning because all ordering changes (except to ascending or descending) will have no adverse effects on the database.

A.1.2 Errors

A compilation error has the following form: 

dbrev fatal compile error: message

Individual compilation error messages are listed below.  Each message appears in an example box and is followed by an explanation of its meaning.  

destination record type... recordname is illegal member of project of recordname

A record type named in a project record statement contains fields from a source record type other than the one named in the statement.  

destination recordname involved in project and join

A record type named in a join record statement contains fields from a source record type other than those named in the statement.  

destination record type recordname, member of set setname, 
must be derivable from a member of the combined sets in the source database

Destination set setname has been named in a combine set statement.  Record type recordname is a member, in the destination database, of setname.  But for the combine statement to work, all destination member record types must be derivable from records in the source, which are members of the sets being combined.  

destination record type recordname must be included in 
'project record' statement for recordname

A project record statement has been used, but record type recordname was not listed in the statement.  However, the record type has been derived to be a member of the same projection.  

destination record type recordname needs 'join record' statement

Destination record type recordname contains fields from multiple source record types, but has not been named in a join record statement.  

errors in command-line script

The statement(s) on the dbrev command line (following the -e option) contain syntax errors.  (The most common error is the absence of a semicolon at the end of the script.)  

errors in RDL file

The RDL file contains syntax errors. 

illegal join: non-tail record type recordname cannot own set type setname

Source record type recordname is named in a join record statement as a non-tail record.  This means that it may be replicated two or more times in the destination database.  (See the join record statement in section 3.4.1, "Record Changes.")  Because of the potential replication, the destination record type cannot be made the owner of an existing set type (except for one owned by the tail record type).  

illegal split of set setname into setname: destination 
member record type recordname not derivable from source

A split set statement names a multiple-member source set type and two or more destination set types.  Each member of the destination set types must be derivable from a member of the source set type.  

initialized record type recordname cannot be target of join

Record type recordname has been named in a join record statement and an initialize record statement.  This would result in the loss of all data from all records named in the join.  Eliminate one of the statements.  

initialized record type recordname cannot be target of project

Record type recordname is the target of a project operation and has also been named in an initialize record statement.  This can cause the inadvertent loss of data.  If this is the desired action, the destination field types should be statically renamed before the revision, then changed back after the revision.  

initialized record type recordname must not be contained in any 
non-empty sets (setname)

When a record type is initialized, any sets it is either owner or member of must be either deleted or initialized.  

line number: error limit

In the RDL file, more than 10 syntax errors have been detected.  

line number missing brace (})

A syntax error in the RDL file. 

line number missing quote

A syntax error in the RDL file.  

line number out of memory

During compilation of the RDL, temporary memory is allocated for building lists.  This error indicates that the calloc function has not been able to allocate more memory.  You may need to delete resident programs (on DOS) or increase memory on  your computer.  

line number: premature EOF: probably missing '*/'

The end of the RDL file that did not follow a statement-terminating semicolon has been located.  

member of normalized set setname (recordname) must be product of a project operation

A normalize set statement has named a set that is not between two records that are components of the same projection from one source record.  

out of memory

This error indicates that the calloc function was unable to allocate more memory. You may need to delete resident programs (on DOS) or increase memory on your computer.  

owner of normalized set setname (recordname) must be product of a project operation

A normalize set statement has named a set that is not between two records that are components of the same projection from one source record.  

record type recordname is not legal owner of set type setname

A split set statement has named set type setname as a destination set type.  All destination set types listed in the statement must have an owner record type that is derivable from the same source owner record type.  

set type setname mismatching members

A set type with the same name and owner record type in both source and destination databases contains no destination members that are derived from source members.  

set type setname mismatching owners: source recordname, destination recordname

The owner record type of a set type in the destination database cannot be derived from the owner of the same set type in the source database.  

source record type recordname involved in project and join

Source record type recordname is named in a join record statement, but its fields are found in destination record types other than the one named in the statement.  

source record type recordname needs 'join record' statement

Source record type recordname has not been named in a join record statement, but some of its fields are included in a joined record.  

source record type recordname, member of set setname, has no corresponding member 
type in destination set setname

A source set type that is named in a combine set statement contains a member record type, which is not included in the combined destination set.  

unable to open RDL file filename

The RDL file, as named on the command line (following the -s option) cannot be opened.  It may be the wrong name or may not have read permission.  The file name must be the complete absolute or relative path name of the file.  

unable to open report file filename

The report file name is created from the source database path name, with a .rep suffix added.  If the report file cannot be opened, there may be some operating system restriction on the directory.  Check permissions or directory size.  

unable to open temporary file

When a command line script is used, it is placed into a temporary file that is named dbr followed by a number (e.g., dbr8776).  This error will occur if the file cannot be created.

A.2 Execution Phase

Warnings are provided during the execution phase of db_REVISE when an action is not able to be carried out or when some data is lost.  Execution warnings will be printed by default.  The -x option can be used to suppress the printing of warnings (i.e., -x0).  By default, db_REVISE will continue to execute if there are warnings.  The -a option (abort level) can be used to request termination if there are any execution warnings (i.e., -a1).

Errors during the execution phase will cause db_REVISE to close the databases and terminate.

A.2.1 Warnings

An execution warning has the following form: 

dbrev execution warning: message File: filename, Record type: recordname, 
Record number: number

Individual execution warning messages are listed below.  Each message appears in an example box and is followed by an explanation of its meaning.  

field conversion for value too long, truncated

A convert field statement using a printf format has created an ASCII string too long to fit into the destination field.  

Not enough value of fieldname in destination; truncated from 'value' to value

While a numeric field was converted into an ASCII string, the program discovered that the string was not long enough to hold the entire result.  

not enough room for value of fieldname in destination; truncated from 
'value' to value

While an ASCII string was converted into a numeric value, the program discovered that the full value could not be stored in the numeric field.  

not enough room for value of fieldname in destination; truncated from 
'value' to value

A numeric value has been shortened in the destination database (e.g., from long to int) and the full value of the number cannot be stored in the destination.

A.2.2 Errors

An execution error has the following form: 

dbrev fatal execution error: message File: filename, Record type: recordname, 
Record number: number

For example, if db_REVISE printed the following message: 

dbrev fatal execution error: duplicate key File: tims.d02, Record type: INFO, 
Record number: 43

This is an indication of a database corruption in the source database. See the dbcheck utility for how to determine about the corruption and possible remedies.

Individual execution warning messages are listed below.  Each message appears in an example box and is followed by an explanation of its meaning. 

database error # number

This message will be printed whenever any RDM error is generated in the source, working, or destination database.  The interpretation of the codes is found in the Return Codes section.  (A common error that appears indicates that the destination database files do not exist.  In this case, you should first run initdb on the destination database.) 

delete record out of sequence, turn off -q

An inconsistency has been found in the database address sequence in a file.  Assumptions are made about this sequence to perform optimization of the revision.  To turn off the optimization, do not use the -q command line option. 

duplicate key

During a d_fillnew call to the destination database, the S_DUPLICATE status was returned.  There are several reasons this can occur.  The most likely problem is that a non-unique, or non-key field has been changed into a unique key field.  Other possibilities include: the initialize field statement was used on a unique key field, the field was shortened, or a conversion was done that changed two formerly different values into the same value. 

internal error

There is an apparent software error in db_REVISE that should be reported to Raima Technical Support.

d_lock

Lock group of record or set types

Prototype

int32_t d_lock(
    int32_t             count, 
    const LOCK_REQUEST *lrpkt, 
    DB_TASK            *task, 
    int32_t             dbn);

Parameters

count (input) The number of items to be locked.
lrpkt (input) A pointer to a lock request packet array.
task (input) A pointer to a DB_TASK structure.
dbn (input) The database number.

Description

Function d_lock provides a grouped lock request capability whereby more than one record and/or set type can be locked in a single request. Two parameters are passed to d_lock. The first parameter is an integer that contains a count of the number of items to be locked. The second parameter is a pointer to a lock request packet array. Each lock request is placed in a LOCK_REQUEST structure entry as defined below and declared in file rdm.h.

typedef struct {
  	uint32_t item;  /* record or set type to be locked */
  	int32_t  type;  /* type of lock */
} LOCK_REQUEST;

Structure field item contains the record or set type of the record or set to be locked. Field type contains the type of lock to be applied as follows.

  • 'r' (read lock)
  • 'w' (write lock)
  • 'k' (keep lock)

Read locks allow other DB_TASKs to read but not update the locked files. Write locks prevent all other DB_TASKs from accessing the locked files and can only be issued from within a transaction. Keep locks can only be issued from within a transaction and will keep read-locked and write-locked files read-locked after a transaction ends (d_trend and d_trabort normally frees all read and write-locked files). The record or set types specified with a keep lock must be locked in order to apply a keep lock. Otherwise, status S_NOTLOCKED is returned.

Within a transaction, a read-locked file can be upgraded to a write or exclusive lock by calling d_lock with a write lock where a read lock already exists. If the upgrade lock request is denied, the read lock will remain in effect. However, the denied upgrade request means that another task has made the upgrade request first, and will time out if the read lock is not freed. Thus, upgrade denial should be followed by re-starting the transaction or by freeing all read locks.

If any of the requested locks cannot be granted, none of the locks will be granted.

If a single d_lock call for each transaction is always used to lock all of the files needed by the transaction, and the timeout value is not -1, the program is guaranteed to be free of deadlock.

Return Codes

Value Name Description
0 S_OKAY normal return, okay
5 S_UNAVAIL database file currently unavailable
-11 S_BADTYPE invalid lock value
-23 S_NOTFREE illegal attempt to lock locked set/record
-24 S_NOTLOCKED attempt to access unlocked set/record
-31 S_NOTRANS cannot update database outside a transaction
-33 S_STATIC locks not allowed on static files
-40 S_INVNUM invalid record or set number
-79 S_INROT Cannot apply a write or exclusive lock while in a read-only transaction
-939 S_READONLY unable to update file due to READONLY option

Example

LOCK_REQUEST trx_ckin_pkt[] =
{
    { AUTHOR_LIST,       'r' },
    { HAS_PUBLISHED,     'w' },
    { BORROWER,          'w' }
};
int32_t trx_ckin_sz = sizeof(trx_ckin_pkt) / sizeof(trx_ckin_pkt[0]);
...
d_trbegin("trx_ckin", task);
if ( d_lock(trx_ckin_sz, trx_ckin_pkt, task, CURR_DB) == S_OKAY )
{
    ...                   /* check in borrowed book */
    d_trend(task);
    }
else
    d_trabort(task);      /* try again later */
